# ScopeParity brand and UX system

Last reviewed: 2026-07-18

## Name

**ScopeParity** names the actual job: bringing the scopes used in code, declared for review, explained on public surfaces, and shown in the demo into parity.

Preliminary collision screening found no exact SaaS/developer-tool match, no npm package for `scopeparity` or `scope-parity`, and no `.com` registration record on 2026-07-18. This is naming due diligence, not a trademark opinion.

## Position

ScopeParity is an independent launch instrument, not an authority. The metaphor is a careful preflight table where every claim has a corresponding piece of evidence.

Primary line:

> See the technical story Google will compare, before you submit it.

Supporting line:

> Match runtime scopes, consent-screen values, public identity, and demo evidence—locally, without Google credentials.

## Personality

- **Exact:** names surfaces, files, and rule evidence instead of assigning a vague score.
- **Calm:** treats launch risk as a finite reconciliation task, not a crisis.
- **Independent:** links to Google sources while making no endorsement claim.
- **Private:** makes local execution visible rather than burying it in a footer.
- **Honest:** says “manual confirmation” or “outside ScopeParity” where software cannot know.

Avoid cheerleading, fear-based copy, and bureaucratic language.

## Vocabulary contract

Use:

- in parity / mismatch found;
- objective blocker;
- confirm manually;
- evidence complete;
- submission manifest;
- technical preflight;
- restricted-scope boundary;
- source and ruleset version.

Never use:

- guaranteed approval;
- verified by ScopeParity;
- compliance score;
- audit certificate;
- Google-approved or Google partner;
- legally compliant;
- safe to submit as an absolute claim.

## Visual direction

The product should resemble a well-lit launch room, not a generic neon developer dashboard.

- Foundation: warm paper and graphite rather than pure white/black.
- Structure: thin rules, alignment marks, and one continuous comparison rail.
- Accent: a restrained cobalt for actions and data paths.
- State colors: oxide red for objective mismatch, ochre for manual confirmation, mineral green for evidence present. Never rely on color alone.
- Type: system sans for prose and system mono for scopes, paths, rule IDs, and commands. No remote fonts.
- Motion: one short reconciliation transition at most; respect `prefers-reduced-motion`.
- Density: one dominant artifact per section. Avoid a grid of interchangeable feature cards.

## Interface hierarchy

Every result follows the same reading order:

1. **What differs** — two named surfaces and their values.
2. **Why it matters** — the mechanical consequence, without approval claims.
3. **What to do** — one narrow action.
4. **Why ScopeParity says this** — official source, rule ID, and ruleset date.
5. **What remains unknown** — manual judgment or external assessment.

## Trust surfaces

The landing page and report must both show:

- “Runs locally” adjacent to the command, not only in privacy copy;
- the exact file-discovery boundary (`git ls-files`);
- values the tool never asks for;
- an example report containing paths and line numbers but no source lines;
- the restricted-scope stop state;
- a direct link to the source for every critical rule;
- a plain statement that final review belongs to Google.

## CTA hierarchy

1. Primary: **Run the local scan**
2. Secondary: **Open a real sample report**
3. Commercial: **Compile the launch evidence workspace — ¥59,800 / $399**

Do not lead with checkout before the user can inspect the scanner and report quality.
